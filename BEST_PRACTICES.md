# Terraform Best Practices — This Project

Opinionated engineering standards for this Terraform module library. These rules reflect real-world patterns for scalable, multi-team, multi-account AWS infrastructure.

---

## Project Layers

This project is organized into four layers with strict responsibilities:

```
bootstrap/        # One-time account setup (OIDC trust, S3/DynamoDB state backend)
modules/          # Single-responsibility, reusable Terraform modules (one AWS service each)
stacks/           # Pre-composed module bundles for common workload patterns
environments/     # Per-environment Terraform roots (dev, qa, prod) — thin orchestration only
```

Each layer has a clear contract with the next:
- `bootstrap` creates the foundation that `environments` depends on (state backend, OIDC role)
- `modules` are the atomic units — they know nothing about stacks or environments
- `stacks` compose modules into workload patterns — they know nothing about environments
- `environments` instantiate stacks with environment-specific variable values — no resource logic lives here

---

## Module Standards

Every module must contain exactly:

```
modules/<name>/
  main.tf        # resource definitions only
  variables.tf   # all inputs — every variable needs description + type
  outputs.tf     # all outputs — every output needs description
  README.md      # generated by terraform-docs (run /docs after any variable/output change)
  examples/      # at least one working example (examples/basic/main.tf)
```

Rules:
- Single responsibility: one module, one AWS service or concern
- No environment-specific logic inside modules
- No hardcoded account IDs, region strings, or ARNs — always use variables or data sources
- Use `for_each` over `count` whenever iterating over resources
- All outputs must be stable — downstream stacks depend on them

---

## Stack Standards

A stack composes two or more modules to deliver a deployable workload pattern. Stacks live in `stacks/<name>/` and follow the same file structure as modules.

Rules:
- Stacks wire module outputs to module inputs — no inline `resource` blocks except where a module does not exist
- Stacks publish their key outputs to SSM Parameter Store so downstream stacks can read them without state file access
- Stacks never hardcode values that belong in `environments/` variable files

## Cross-Stack Data Sharing

Modules must not directly depend on each other's internal state. Use one of these controlled patterns:

### Pattern 1 — Direct Output Passing (same root module)

```hcl
module "network" {
  source = "../../modules/network"
  # ...
}

module "ecs" {
  source = "../../modules/ecs"
  vpc_id = module.network.vpc_id   # direct reference — safe within one state
}
```

Use when: both modules are composed inside the same stack and share a single Terraform state.

### Pattern 2 — Remote State Data Source (cross-state, same team)

```hcl
data "terraform_remote_state" "platform" {
  backend = "s3"
  config = {
    bucket = "myapp-terraform-state"
    key    = "prod/platform/terraform.tfstate"
    region = "us-east-1"
  }
}

vpc_id = data.terraform_remote_state.platform.outputs.vpc_id
```

Use when: teams share a state bucket and the consuming stack is tightly coupled to a specific upstream stack. **Prefer SSM (Pattern 3) instead** — remote state couples you to the state file structure and requires backend access permissions across stacks.

### Pattern 3 — SSM Parameter Store (recommended for cross-stack wiring)

**This is the default pattern for this project.** Upstream stacks write outputs to SSM; downstream stacks read them via `data "aws_ssm_parameter"`.

Write (inside the upstream stack's `main.tf`):

```hcl
resource "aws_ssm_parameter" "vpc_id" {
  name  = "/${var.project}/${var.environment}/infra/vpc_id"
  type  = "String"
  value = module.network.vpc_id
}
```

Read (inside a downstream stack or environment root):

```hcl
data "aws_ssm_parameter" "vpc_id" {
  name = "/${var.project}/${var.environment}/infra/vpc_id"
}

module "ecs" {
  source = "../../modules/ecs"
  vpc_id = data.aws_ssm_parameter.vpc_id.value
}
```

SSM naming convention for this project:

```
/{project}/{environment}/infra/{output_name}
```

Examples:
```
/myapp/prod/infra/vpc_id
/myapp/prod/infra/private_subnet_ids
/myapp/prod/infra/ecs_cluster_arn
/myapp/prod/infra/rds_endpoint
```

### Choosing a Pattern

| Scenario | Pattern |
|---|---|
| Two modules in the same stack | Direct output passing |
| Cross-stack, same-team, tightly coupled | Remote state (with caution) |
| Cross-stack, decoupled, or multi-team | SSM Parameter Store (preferred) |
| Multi-account | SSM Parameter Store (IAM-controlled access) |

## State Management

- **Remote backend:** S3 + DynamoDB per environment account (provisioned by `bootstrap/state-backend`)
- **Partial backend config:** `environments/<env>/main.tf` declares an empty `backend "s3" {}` block; actual values are supplied at `terraform init` time via `-backend-config=config/backend-<env>.hcl`
- **State isolation:** one S3 bucket per account, different stacks use different `key` paths
- **Locking:** DynamoDB table prevents concurrent applies to the same state
- **State versioning:** enabled on the S3 bucket for recovery

Never put state bucket names, lock table names, or backend config in `terraform.tfvars` — they belong in `config/backend-<env>.hcl` (committed, no secrets).

## Secrets

| Secret type | How to handle |
|---|---|
| DB passwords | RDS-managed rotation (`master_password_secret_arn = null`) |
| API keys, tokens | `secrets-manager` module; populate value out-of-band |
| TLS certificates | `acm` module with DNS validation |
| IAM credentials | OIDC — no IAM user access keys anywhere |
| CI/CD role ARNs | GitHub Environment secret (`AWS_ROLE_ARN`) — environment-scoped, not repo-scoped |

Rules:
- Never put secrets in `.tfvars` files or environment variables committed to git
- Use `SecureString` type for any SSM parameter that contains a sensitive value
- `secrets.auto.tfvars` is for CI-injected secrets only — it is gitignored and never committed

---

## IAM

- No wildcard (`*`) actions in IAM policies unless explicitly justified with a comment explaining why
- Every Lambda function, ECS task, and EC2 instance gets a **dedicated** execution role — no shared roles across services
- All CI/CD (both Terraform pipelines and application deploy pipelines) uses OIDC assume-role — no IAM users with long-lived access keys
- Two distinct OIDC roles exist per environment:
  - `{project}-github-actions-{env}` — created by `bootstrap/oidc`, used by `tf-plan`/`tf-apply` (Terraform CI)
  - `DevOpsRole` — created by `stacks/platform`, used by application deploy workflows (ECR push, ECS update, Secrets read)

## Resource Naming

All resources follow:

```
{project}-{environment}-{region_short}-{resource_type}[-{qualifier}]
```

Region short codes: `use1` (us-east-1), `use2` (us-east-2), `usw2` (us-west-2), `euw1` (eu-west-1)

Examples:
- `myapp-prod-use1-vpc`
- `myapp-prod-use1-alb-public`
- `myapp-prod-use1-fn-order-processor`

## Required Tags

All modules apply these tags automatically. Never omit them:

```hcl
tags = {
  Project     = var.project
  Environment = var.environment
  Region      = var.region
  ManagedBy   = "terraform"
  Repository  = var.repository
}
```

Add team/cost-center tags via the module's `tags` input variable — do not edit the base tags block inside modules.

## Code Quality

- Run `/lint` before opening any PR (`terraform fmt` + `tflint` + `checkov` + `terraform validate`)
- Run `/docs` after adding or changing any variables or outputs in a module
- No hardcoded account IDs, region strings, or ARNs — use variables or data sources
- Every module must be self-contained: no implicit dependencies on external state except via explicit `data` sources
- All secrets go through Secrets Manager or SSM SecureString — never in `.tfvars`

## Deployment Order (Fresh Environment)

1. `bootstrap/state-backend` — S3 bucket + DynamoDB for Terraform state
2. `bootstrap/oidc` — OIDC provider + Terraform execution role
3. `stacks/platform` — VPC, ECS cluster, ALBs, security groups, IAM, KMS, TLS → writes SSM parameters
4. `stacks/data-layer` — Aurora, ElastiCache, DynamoDB → reads VPC/subnet SSM params from platform
5. Application stacks (`bff`, `microservice`, `serverless`, etc.) → consume SSM parameters from platform and data-layer

---

## General Principles

- **Loose coupling between stacks** — SSM parameters are the contract, not state file structure
- **Strong module contracts** — every input and output is typed and documented
- **Least privilege IAM** — scoped to the minimum needed, never admin unless bootstrapping
- **Plan before apply** — always review `terraform plan` output; CI enforces this on PRs via `tf-plan.yml`
- **Pin versions** — all modules pin `required_providers` versions; environments pin the Terraform binary version
- **No resources in environments** — environment roots call stacks/modules; they do not define `resource` blocks directly
